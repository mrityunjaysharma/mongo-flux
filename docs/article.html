The Problem

If you've ever run a $group aggregation on a MongoDB collection with millions of documents, you know the pain. A simple "revenue by region" query that should take milliseconds ends up taking seconds. Your dashboard times out. Your data team starts asking about Snowflake.

The root cause is architectural: MongoDB stores data row-by-row. When you ask "what's the average order amount per region?", it reads every field of every document — even though you only need two columns. That's the fundamental mismatch between OLTP and OLAP workloads.

OLTP (MongoDB)OLAP (ClickHouse)
StorageRow-oriented (full documents)Column-oriented (only needed columns)
Compression~2x~10-20x (columnar + codecs)
GROUP BY 500K rows300-1500 ms8-30 ms
Best atPoint lookups, writesAggregations, scans

The traditional fix is ETL: batch jobs that copy data to a warehouse every hour. But ETL means stale data, pipeline maintenance, and schema drift headaches.

We built MongoFlux to eliminate ETL entirely.

The Solution: ClickHouse as a "Virtual Secondary"

MongoDB replica sets work by having secondary nodes tail the primary's oplog — a capped collection recording every write. MongoFlux does exactly the same thing, but applies operations to ClickHouse instead of a local storage engine.

The default deployment runs a 3-node MongoDB replica set: 1 primary and 2 secondaries. For reads, the application connects to MongoFlux which acts as a routing proxy — if ?clickhouse=true is in the URI, the query goes to ClickHouse. Otherwise, it's forwarded to MongoDB Primary unchanged.

flowchart TD APP[Your Application] -->|writes| MGP[(MongoDB Primary)] MGP -->|replication| MGS1[(Secondary 1)] MGP -->|replication| MGS2[(Secondary 2)] APP -->|"all reads"| MGC[MongoFlux] MGC -->|"?clickhouse=true → SQL"| CH[(ClickHouse)] MGC -->|"no flag → forward"| MGP MGP -->|"oplog (async)"| MGC MGC -->|batch INSERT| CH
Key insight: Writes go to the MongoDB primary unchanged (zero overhead). The two secondaries provide HA. MongoFlux routes reads — analytics queries to ClickHouse, everything else forwarded to MongoDB. Add ?clickhouse=true to your connection URI. No application code changes.

How It Works

1. Real-Time Replication

MongoFlux tails the oplog from the MongoDB primary. The two secondaries handle read scaling and failover independently.

sequenceDiagram participant App participant Primary as MongoDB Primary participant S1 as Secondary 1 participant S2 as Secondary 2 participant mf as MongoFlux participant CH as ClickHouse App->>Primary: insertOne({amount: 99.99}) Primary-->>App: acknowledged ✓ Primary->>S1: replication (async) Primary->>S2: replication (async) Note over App: Write done. Zero overhead. Primary->>mf: oplog entry (async) mf->>mf: extract fields → batch mf->>CH: INSERT batch mf->>mf: save position

The replication is fully async. The primary acknowledges writes before MongoFlux sees them. If the primary fails, the replica set elects a new primary and MongoFlux automatically reconnects. Benchmarks confirm zero measurable write overhead.

2. Query Translation (BSON → AST → SQL)

MongoDB aggregation pipelines are translated through a two-phase expression tree:

flowchart LR A["MongoDB Pipeline
{$group: {_id: '$region',
total: {$sum: '$amount'}}}"] -->|parse| B["Expression Tree
(AST)"] B -->|emit| C["ClickHouse SQL
SELECT region, sum(amount)
FROM orders GROUP BY region"]

The translator supports 12 pipeline stages, 12 accumulators, and 40+ expressions — arithmetic, string, date, conditional, array, and type operations.

3. Crash Recovery

stateDiagram-v2 [*] --> Tailing Tailing --> Flushing: batch ready Flushing --> Tailing: OK → save position Flushing --> Tailing: FAIL → keep rows Tailing --> [*]: SIGTERM → flush + save

Oplog position is saved only after successful flush. Crash before save → replay from last position. ReplacingMergeTree deduplicates replays automatically. At-least-once delivery guaranteed.

Standalone vs Distributed ClickHouse

MongoFlux auto-generates the correct DDL for both modes. For clustered deployments, it creates a local MergeTree table on each shard plus a Distributed routing table — all from a single mapping configuration.

ModeWhen to useAvg speedup vs MongoDB

Distributed is slower at small data sizes due to cross-shard coordination overhead. It wins at 10M+ rows per shard where parallel scan outweighs network cost.

Getting Started

git clone https://github.com/your-org/MongoFlux
cd MongoFlux
docker compose up --build

This starts a 3-node MongoDB replica set (1 primary + 2 secondaries on ports 27017-27019), ClickHouse, and MongoFlux. The replica set initializes automatically via rs.initiate().

# Create a mapping
curl -X POST http://localhost:9090/api/v1/mappings -H "Content-Type: application/json" -d '{
  "collection": "orders",
  "clickhouse_table": "orders",
  "clickhouse_database": "analytics",
  "fields": [
    {"mongo_field": "_id", "ch_column": "id", "ch_type": "String"},
    {"mongo_field": "amount", "ch_column": "amount", "ch_type": "Float64"},
    {"mongo_field": "region", "ch_column": "region", "ch_type": "LowCardinality(String)"}
  ],
  "engine": "MergeTree",
  "order_by": ["id"]
}'

# Sync the table
curl -X POST http://localhost:9090/api/v1/mappings/orders/sync

Insert data into MongoDB normally. Query it from ClickHouse at 27x the speed. No ETL. No application changes.

When Should You Use This?

Good fit: MongoDB in production + slow aggregations + need real-time analytics without ETL complexity.

Not a fit: Simple point lookups (MongoDB is already fast), need sub-millisecond replication, or strong consistency on deletes.
