The Problem

I had a MongoDB database with a few million documents. The OLTP workload was fine — point lookups, single-doc writes, transactions all ran fast. But the moment someone asked "what's our revenue by region this quarter?" or "show me the top 10 customers by spend," everything fell apart.

A simple $group over 500K documents took 300–1500ms. That's not a dashboard. That's a loading spinner.

The standard solutions all have the same shape: build an ETL pipeline, maintain a separate data warehouse, accept stale data, and split your application code into two paths — one for writes, one for reads. Two connection strings. Two query languages. Two systems to monitor.

I didn't want any of that. I wanted one interface for everything. Same find(), same aggregate(), same MongoDB query syntax. Just faster.

The Idea

MongoDB already has a mechanism for real-time replication: the oplog. Every write to the primary gets recorded in local.oplog.rs, and secondaries tail it with a cursor to stay in sync. What if I built something that tails the oplog the same way — but instead of replicating to another MongoDB node, it replicates to ClickHouse?

ClickHouse is a columnar database. It's designed for exactly the queries MongoDB is bad at: full-table scans, GROUP BY over millions of rows, percentiles, time-range aggregations. A query that takes 565ms in MongoDB takes 4ms in ClickHouse. That's not a typo.

The insight was simple: make ClickHouse behave like a "virtual secondary" in the replica set. Same write stream, zero overhead on the write path, and a query router that transparently sends analytics to ClickHouse while forwarding point lookups to MongoDB unchanged.

How MongoFlux Works

Three things happen:

Writes go directly to MongoDB. MongoFlux is not in the write path at all. Your application writes to the primary exactly as before. MongoDB acks the write before MongoFlux even sees it.

Replication — MongoFlux opens a tailable-await cursor on local.oplog.rs (the same mechanism secondaries use), extracts the fields you've mapped, batches them into groups of 1000, and flushes to ClickHouse via HTTP every 500ms. It persists oplog timestamps to disk so it can resume after a crash without data loss.

Reads — your app sends queries to MongoFlux. If the URI has ?clickhouse=true, the query gets translated from MongoDB's BSON format to ClickHouse SQL and executed there. Otherwise it's forwarded to a MongoDB secondary unchanged. The routing decision happens at the very first step — no unnecessary processing for passthrough queries.

The key constraint: Zero write overhead. If MongoFlux adds even 1ms to write latency, it's not worth it. The oplog tail is fully asynchronous — it's a reader, not a participant in the write path.

The Query Translator

This was the most interesting engineering challenge. MongoDB queries are expressed as BSON documents — nested, operator-heavy, with a completely different mental model than SQL. I needed to translate them faithfully.

The translator uses a two-phase AST approach:

Phase 1: Parse. The BSON query (whether it's a find() filter or an aggregate() pipeline) gets parsed into an internal expression tree. Each node represents an operation — a comparison, a function call, a logical connector, a column reference.

Phase 2: Emit. The expression tree gets walked to produce ClickHouse SQL. Column names are resolved through the schema registry (which maps MongoDB field names to ClickHouse column names), and MongoDB-specific operators get translated to their ClickHouse equivalents.

// Input: MongoDB aggregate pipeline
[{$group: {_id: "$region", total: {$sum: "$amount"}}}]

// Phase 1: Expression tree
GroupNode {
  key: ColumnRef("region"),
  accumulators: [
    Accumulator("sum", ColumnRef("amount"), alias: "total")
  ]
}

// Phase 2: ClickHouse SQL
SELECT region, sum(amount) AS total FROM orders GROUP BY region

The translator handles 12 pipeline stages, 12 accumulators, and 40+ expressions covering arithmetic, string manipulation, date extraction, and conditionals. It's not a full MongoDB compatibility layer — it covers the patterns that matter for analytics.

The Oplog Sync Engine

The sync engine is where reliability matters most. It needs to handle:

Crash recovery. The oplog position is saved only after a successful flush to ClickHouse. If MongoFlux crashes between receiving an oplog entry and flushing it, the entry will be replayed on restart. ClickHouse's ReplacingMergeTree engine handles deduplication, so replays are safe. This gives us at-least-once delivery semantics.

Backpressure. If ClickHouse is slow or temporarily unavailable, the sync engine buffers rows in memory up to a configurable threshold (max_pending_rows). Once the threshold is hit, it stops reading from the oplog until the buffer drains. This prevents OOM without losing data — the oplog is still there, we just pause reading it.

Partial updates. MongoDB's oplog records partial updates as $set operations, not full documents. For these, the sync engine fetches the current full document from MongoDB and inserts that into ClickHouse. This ensures ReplacingMergeTree always has the latest complete state.

// The core loop (simplified)
for cursor.Next(ctx) {
    entry := decode(cursor)
    
    switch entry.Op {
    case "i": // INSERT
        batch.Add(extractFields(entry.Doc, mapping))
    case "u": // UPDATE  
        if isFullReplacement(entry.Doc) {
            batch.Add(extractFields(entry.Doc, mapping))
        } else {
            fullDoc := fetchFromMongo(entry.ID)
            batch.Add(extractFields(fullDoc, mapping))
        }
    case "d": // DELETE
        if config.PropagateDeletes {
            batch.Add(tombstoneRow(entry.ID))
        }
    }
    
    if batch.Full() || timeElapsed > 500ms {
        clickhouse.InsertBatch(batch)
        saveOplogPosition(entry.Timestamp)
    }
}

Schema Registry

Not every MongoDB field needs to go to ClickHouse. The schema registry lets you define exactly which fields to replicate and how to map them:

{
  "collection": "orders",
  "clickhouse_table": "orders",
  "clickhouse_database": "analytics",
  "fields": [
    {"mongo_field": "_id",        "ch_column": "id",         "ch_type": "String"},
    {"mongo_field": "amount",     "ch_column": "amount",     "ch_type": "Float64"},
    {"mongo_field": "status",     "ch_column": "status",     "ch_type": "LowCardinality(String)"},
    {"mongo_field": "created_at", "ch_column": "created_at", "ch_type": "DateTime"}
  ],
  "engine": "ReplacingMergeTree",
  "order_by": ["created_at", "id"]
}

The registry is thread-safe, supports hot reloading via the management API, and generates ClickHouse DDL automatically — including distributed table creation for multi-shard deployments.

The break-even point

The first question: at what data size does MongoFlux become faster than MongoDB for aggregations?

The crossover is around 5–10K documents. Below that, MongoDB wins because of HTTP round-trip overhead to ClickHouse. Above that, MongoFlux wins and the gap keeps growing — columnar scans are O(columns touched), not O(rows scanned).

Adding indexes to MongoDB doesn't help here. Indexes are designed for point lookups and range scans on indexed fields, not full-collection aggregations.

Architecture Decisions

Why oplog tailing instead of Change Streams?

Change Streams are the "official" way to watch for changes in MongoDB. But they add overhead: they go through the MongoDB query engine, require a $match stage, and have higher latency. Oplog tailing is what secondaries actually use — it's the lowest-latency, lowest-overhead path to get changes. MongoFlux supports both modes (for Atlas/sharded clusters where direct oplog access isn't available), but oplog mode is the default.

Why HTTP for ClickHouse instead of the native protocol?

ClickHouse's HTTP interface is simpler, doesn't require a C++ client library, and supports the same batch INSERT syntax. The overhead of HTTP vs native protocol is negligible compared to the actual query execution time. And it means MongoFlux has zero C/C++ dependencies — it's pure Go.

Why ReplacingMergeTree?

Because of at-least-once delivery. If MongoFlux crashes after inserting a batch but before saving the oplog position, it will replay those rows on restart. ReplacingMergeTree deduplicates rows with the same primary key during background merges. This gives us exactly-once semantics without requiring exactly-once delivery — a much simpler engineering constraint.

Why not a wire protocol proxy?

The initial design considered implementing the MongoDB wire protocol so MongoFlux could sit transparently between the app and MongoDB. I rejected this because: (1) it's a massive compatibility surface, (2) it adds latency to every operation including writes, and (3) the ?clickhouse=true parameter approach is explicit and debuggable. You always know which path a query took.

Production Considerations

Observability. MongoFlux exposes Prometheus metrics at /metrics: rows synced per collection, flush success/failure counts, oplog lag, pending buffer size, and flush duration. You can alert on mongoflux_oplog_lag_ms > 5000 to catch replication falling behind.

Graceful shutdown. On SIGTERM, MongoFlux flushes all pending batches, saves the oplog position, and exits cleanly. No data loss on deploys.

Distributed ClickHouse. For multi-shard deployments, MongoFlux generates both the local table (on each shard) and the Distributed table automatically with ON CLUSTER DDL. You just add "cluster": "prod" and "sharding_key": "cityHash64(user_id)" to your mapping.

Failure modes. MongoFlux crash → resume from saved position (no data loss). ClickHouse down → retry on next flush (rows buffered in memory). MongoDB primary failover → auto-discovers new primary with 3s backoff. The only scenario with potential data loss is a corrupted oplog position file, which would cause MongoFlux to start from the oplog tail (missing the gap between the corrupted position and current tail).

What I'd Do Differently

Initial bulk sync. Right now, MongoFlux only captures changes from the moment you start it. If you have existing data, you need to backfill it separately. A built-in bulk sync mode is the highest-priority item on the roadmap.

Partial update handling. Fetching the full document on every $set update adds load to MongoDB. A smarter approach would be to maintain a local cache or use ClickHouse's ALTER UPDATE for simple field changes. This is complex to get right with ReplacingMergeTree semantics.

HA. Currently MongoFlux is a single process. For production, you'd want leader election so a standby can take over if the primary crashes. This is straightforward with etcd or a simple MongoDB-based lock, but it's not built in yet.

Getting Started

git clone https://github.com/your-org/MongoFlux
cd MongoFlux
docker compose up --build

This starts MongoDB (3-node replica set), ClickHouse, and MongoFlux. Create a mapping via the API, insert some data into MongoDB, and query it through ClickHouse. Five minutes from clone to 27x faster queries.

# Create a mapping

  -H "Content-Type: application/json" \
  -d '{
    "collection": "orders",
    "clickhouse_table": "orders",
    "clickhouse_database": "analytics",
    "fields": [
      {"mongo_field": "_id", "ch_column": "id", "ch_type": "String"},

    "engine": "ReplacingMergeTree",

# Create the ClickHouse table
curl -X POST http://localhost:9090/api/v1/mappings/orders/sync

TL;DR

MongoFlux sits beside your MongoDB replica set, tails the oplog like a secondary would, replicates to ClickHouse in real-time, and routes analytical queries there transparently. Your app doesn't change. Writes don't slow down. Analytics go from hundreds of milliseconds to single-digit milliseconds.

It's not magic — it's just using the right storage engine for the right workload, with a sync layer that makes the two systems behave as one.

When to use MongoFlux: You have MongoDB in production, your aggregation queries are slow, and you want real-time analytics without the complexity of ETL pipelines or data warehousing infrastructure.

When not to use it: Your queries are simple point lookups (MongoDB is already fast), you need sub-millisecond replication consistency, or you need strong consistency on deletes.
