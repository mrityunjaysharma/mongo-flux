The Problem We Solved

MongoDB is great for OLTP. Point lookups, single-doc writes, transactions — all fast. But the moment you need analytics (GROUP BY over millions of rows, time-range scans, percentiles), it falls apart. You end up building ETL pipelines, maintaining a separate warehouse, dealing with stale data. And the classic workaround — reading from one system, writing to another — means your application code splits into two paths, two connection strings, two query languages.

MongoFlux eliminates all of that. Your app talks to one place for both reads and writes. No code changes, no query rewrites. You set ?clickhouse=true once in your read connection string and everything else stays the same — same find(), same aggregate(), same MongoDB query syntax. MongoFlux translates it to SQL behind the scenes.

Architecture

MongoDB runs as a 3-node replica set. The application connects to MongoFlux for reads. If the URI contains ?clickhouse=true, MongoFlux translates the query to SQL and sends it to ClickHouse. Otherwise it forwards to MongoDB unchanged. Writes go directly to the primary. MongoFlux tails the oplog asynchronously — making ClickHouse behave like a "virtual secondary."

flowchart TD APP[Your Application] APP -->|"writes (insert/update/delete)"| MGP[(MongoDB Primary)] MGP -->|replication| MGS1[(Secondary 1)] MGP -->|replication| MGS2[(Secondary 2)] MGP -.->|"oplog tail (async)"| SYNC[Oplog Sync Engine] SYNC -.->|"batch INSERT"| CH[(ClickHouse)] APP -->|"reads (find/aggregate)"| ROUTER[Query Router] ROUTER -->|"clickhouse=true"| XLAT[Query Translator] XLAT -->|SQL| CH ROUTER -->|"clickhouse=false"| MGP SYNC -.->|lookup| REG[Schema Registry] XLAT -.->|lookup| REG API[Management API + /metrics] --> REG
Key insight: Writes go to MongoDB unchanged (zero overhead). Secondaries provide HA. MongoFlux routes reads — analytics to ClickHouse, everything else to MongoDB. Your app uses one interface for everything. No split paths, no code changes.

Data Flow

Write Path (zero overhead)

Writes target the MongoDB primary. The two secondaries replicate via the standard protocol. MongoFlux tails the oplog independently — it is not in the write path.

Zero overhead. Primary->>mf: oplog entry (async) mf->>mf: extract fields → batch mf->>CH: INSERT batch (every 500ms) mf->>mf: save position

Same interface for everything. No split paths.

Components

Oplog Sync

Tails local.oplog.rs with tailable-await cursor. Position saved AFTER flush. Failed batches retained for retry. Backpressure via max_pending_rows. Optional delete propagation.

Change Stream Sync (Atlas/Sharded)

Alternative for environments without direct oplog access. Per-collection threads with resume token persistence.

Position saved only after successful flush. Crash before save → replay. ReplacingMergeTree deduplicates. At-least-once delivery guaranteed.

Production Features

  • Prometheus metrics — rows synced, flush latency, oplog lag, buffer size
  • Backpressure — configurable max pending rows prevents OOM
  • Delete propagation — optional tombstone rows for soft-delete tracking
  • Config validation — fails fast with clear errors on startup
  • Graceful shutdown — flushes pending batches, persists position on SIGTERM
  • Kubernetes-ready — non-root container, tini PID 1, health probes

Getting Started

git clone https://github.com/your-org/MongoFlux
cd MongoFlux
docker compose up --build

Starts MongoDB (3-node replica set), ClickHouse, and MongoFlux. Create a mapping, insert data, query at 27x speed. Five minutes.

When Should You Use This?

Good fit: MongoDB in production + slow aggregations + need real-time analytics without ETL complexity.

Not a fit: Simple point lookups (MongoDB is already fast), need sub-millisecond replication, or strong consistency on deletes.
