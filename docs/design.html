1. Overview

mg-clickhouse is a CDC bridge that makes ClickHouse behave as a "virtual secondary" in a MongoDB replica set. It tails the oplog using the same mechanism real secondaries use, translates MongoDB queries to ClickHouse SQL via an expression tree AST, and routes reads transparently based on a single URI parameter.

Design Goals

  • Zero write overhead on MongoDB (async oplog tailing, decoupled from write path)
  • Sub-second replication latency (same as MongoDB secondary nodes)
  • Transparent query routing (single URI parameter switches reads to ClickHouse)
  • Support both standalone and multi-shard ClickHouse deployments
  • Crash recovery with at-least-once delivery semantics
  • No application code changes required

3. Component Design

3.1 Oplog Sync Engine

The core replication component. Mirrors exactly what MongoDB secondaries do internally — opens a tailable-await cursor on local.oplog.rs and applies operations to ClickHouse.

sequenceDiagram participant OS as OplogSync participant MG as MongoDB
local.oplog.rs participant REG as Schema Registry participant CH as ClickHouse OS->>MG: Open tailable-await cursor
(ts > last_saved_position) loop For each oplog entry MG-->>OS: {op: "i", ns: "db.orders", o: {...}} OS->>REG: has_mapping("orders")? REG-->>OS: CollectionMapping OS->>OS: Extract mapped fields → batch alt Batch full OR flush interval elapsed OS->>CH: INSERT INTO table VALUES (batch) CH-->>OS: OK OS->>OS: Save oplog timestamp to disk end end
Critical design decision: Oplog position is saved ONLY after successful flush to ClickHouse. If a crash occurs between flush and save, the same entries are replayed on restart (at-least-once semantics). ReplacingMergeTree deduplicates replayed inserts.

3.2 Query Translation (Expression Tree AST)

Two-phase architecture separating parsing from SQL emission. This enables independent testing, tree-level optimizations, and multiple output targets.

3.3 Schema Mapping Registry

Thread-safe in-memory registry with JSON file persistence. All consumers (oplog sync, query translator, management API) access it through mutex-protected methods.

9. Security

  • ClickHouse credentials URL-encoded (prevents injection via special characters)
  • CURL handles use RAII (no resource leaks on exceptions)
  • Container runs as non-root user (mgch)
  • Management API should be deployed behind API gateway for authentication
  • Config validation rejects invalid values at startup (fail-fast)
